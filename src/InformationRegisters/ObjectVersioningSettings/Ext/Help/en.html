This list is used to manage object versioning settings.

Document and catalog versioning is a useful tool for change history tracking.

To enable object change history, execute the Common settings command located in the Administration section and select the Object versioning check box (selected by default).

This will make the Versioned objects command available in the Administration - Common settings and Integrated subsystems sections. Execute this command to proceed to the versioning settings.

The list includes:

  • Object - list of applied objects displayed as a tree, grouped by object type.
  • Versioning method - the selected versioning method.

Selecting versioning methods

Select one or several objects, execute the Set versioning method command, and select a preferred versioning method for each document and catalog type from dropdown list. This command is also available via the All actions (or More, in Taxi interface) menu, or context menu accessed by right-clicking.

  • No versioning - object version history is not maintained;
  • Versionize when writing - a record is added to the version history whenever a document or catalog is created or modified;
  • Versionize when posting - a new version is saved on posting. Only applicable for documents that can be posted.
  • Default - recommended settings for catalogs (do not versionize) and documents (versionize on post).

Versioning settings can be modified for multiple applied objects of the same type at once. For example, you can change versioning settings for all documents, or for all catalogs.

Practical applications of versioning

  • During the early deployment stages, when data volume is relatively low and human errors (entering incorrect data, mistakenly deleting valid data, etc) are likely, versioning can assist you to keep your documents and lists in order, rapidly eliminate the errors, and educate the application users.
  • Versioning allows you to easily roll any document or catalog item back to an earlier version that does not yet contain any errors or missing data.
  • As data volume tends to increase with time, it is feasible to limit object versioning to the critical operations, such as document posting, or to waive the practice altogether.
  • Versioning for a large number of applied objects requires significant disk space for version storage and deteriorates application performance. We recommend that you use this feature selectively, balancing advantages against performance issues on a case-by-case basis.

Notes on the Versionize on post method

  • If versioning method is set to Versionize on post for a document, no document versions will be saved until the document is posted for the first time. Once the document has been posted, new versions will be saved normally upon any change. This behavior is deliberate; it helps to avoid saving multiple versions of unposted documents, thus saving storage space. We recommend that you apply this setting to all documents.
