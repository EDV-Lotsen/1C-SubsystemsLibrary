Simplified mode for access rights management.

User access rights management is profile-based. A profile defines a set of actions with the application data the user is allowed to perform. Generally, profiles correspond to user job functions or duties. Multiple profiles can be assigned to the user at the same time, these profiles collectively defining their personal access rights settings.

To set up user (external user) access rights, execute the Access rights command located on the navigation panel.

User profiles

In the User profiles list, select check boxes for the profiles required for the user's application activities.

Generally, a profile incorporates several roles. Assigning a profile to user automatically assigns them all roles contained in the profile. For example, the Administrator profile contains the Full rights role granting unlimited access to all data. Only application administrators (users with the Administrator profile) can set up user access rights.

All profiles are broken down into two categories:

  • Primary profiles define a set of access rights sufficient to perform specific activities in the application.

    For example, if a user only performs accounting activities in the application, they should be assigned the Accountant profile (as well as the User technical profile required for application logon).

  • Additional profiles are used to grant users additional rights not covered by the primary profiles. When a primary profile (Accountant, Warehouse manager, etc) does not provide sufficient rights they should be complemented with an additional profile (Counterparty communication, Unposted document printing, etc).

Access kinds

This list is displayed to the right of the profile list, provided that the Use record-level security check box is selected in the Administration - User and rights settings section. The Access kind field is used to precisely define the data area accessible to the user.

An access kind is a rule governing access to application data. For example, you can use the Companies access kind to allow the user to view documents for a specific company only, to block the user from viewing documents for a specific company, etc.

Contents of the access kind list may vary depending on the profile selected in the profile list (to the left). Moreover, profile determines behavior of access kind settings: read-only access to documents, or allow editing. For example, if a profile includes the Product receipt document view role, users with this profile are only allowed to view these documents. However, if the profile includes roles that allow adding and modifying documents, user with this profile will also be allowed to edit these documents.

You need to specify Access values for each access kind. Select All prohibited or All allowed, or specify exceptions. Any access kind can be set up by two methods:

  • specify the allowed value list. We recommend that you use this setup method when you only need to grant the users access to a specific limited data area.
  • specify the prohibited value list. We recommend that you use this method when you need to grant the users access to all data except specific areas. In this case, users automaticaly have access to any new data entered into the application. For example, if you set up the Companies access kind to prohibit the users from accessing documents belonging to Forward Thinking Inc., and then you create a new company (New Tech LLC) in the application, the users will not be prohibited from accessing any documents belonging to New Tech LLC. Please keep in mind that "allow" settings have priority over "prohibit" settings. If users from the above example are allowed to access documents belonging to Forward Thinking Inc. via any other group based on the same profile, they will have access to all documents belonging to this company.
    • To create exceptions, click Add and select the required values from list.

In certain cases, it is expedient to specify item groups (instead of single items) in the prohibited/allowed value list. This facilitates the access rights setup procedure. For example, if you set up the Partners access kind to prohibit access to some of the partners, you can specify partner access groups instead of single partners. For this, you will need to create the list of partner access groups, and assign each partner to a group. These activities are not limited to administrators; any user with sufficient rights to edit the partner list can perform this.

For some profiles, the Users (or External users) access kind can be used for access rights setup. This only applies to the documents (and other applied objects) containing fields that can be used to specify users allowed to access these documents (Responsible, Author, Performer, etc). To grant access to all restricted data available to a specific user, you need to add this user to the allowed value list for the Users access kind. For example, if access to product/service receipt documents is governed by the Responsible field value, document posting will be allowed both for users specified in this field, and users included in the allowed list for the Users access kind. The "prohibit" setting follows the same logic.
