Access groups are used to manage access rights and restrictions for specific users and user groups.

This method allows you to control user access to application functions, lists, and documents. A variety of predefined role sets is available in the application. Moreover, you can modify any access settings manually. Only administrator is allowed to manage user access settings. The user assigned as responsible manager of an access group is allowed to change contents of their access group.

Access rights and restrictions are determined by the access group profile.

For example, you can create an access group named Business software sales managers on the basis of the Sales managers profile, and add two users to this group. The profile supports access restrictions by the Product and service type access kind. Now, if you set all values for this access kind to Prohibited and add Business software to the product type list, both members of the access group will be only allowed access to application data and functionality related to business software.

We recommend that you give your access groups names that reflect the underlying profile's purpose and describe group-level restriction.

Both single users (external users) and user groups (external user groups) can belong to an access group. When working with large numbers of users belonging to multiple access groups, we recommend that you group these users for your convenience. Whenever possible, add user groups, instead of users, to access groups; this helps to minimize regular changes of user lists in multiple access groups.

User role sets

User role sets are detemined automatically, based on the access group the user belongs to (both explicitly and indirectly, via user groups). They are updated whenever lists of users belonging to access groups or user (external user) groups are modified. We recommend that you refrain from editing user role sets by any other methods (for example, via Designer), as these edits will be nullified during the earliest change of access group user list.

Merging access rights

User can belong to multiple access groups simultaneously. In this case, their access rights granted by each access group are merged, according to logical disjunction. For example, let us consider a user belonging to the Acquisition managers group, which grants them access rights to view the Product receipt documents for Forward Thinking Inc. Membership in the Sales managers group also grants them rights to create the Customer order documents for New Tech LLC. Please keep in mind that only access rights are merged as described above; the allowed value lists by group are not merged. Therefore, the user in the above example will not be granted access rights to view the Product receipt documents for New Tech LLC, or to create the Customer order documents for Forward Thinking Inc.

Administrators access group

A predefined access group named Administrators is available for administration purposes, based on the Administrator profile. Any user added to this group is automatically granted the Full rights role. If any user not belonging to the Administrators group was granted the Full rights role, the role will be revoked.

Warning

We recommend that you refrain from using 1C:Enterprise configuration mode for access rights setup purposes.
